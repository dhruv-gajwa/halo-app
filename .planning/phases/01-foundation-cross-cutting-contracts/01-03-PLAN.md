<objective>
Build the namespaced, versioned `localStorage` envelope: a tiny pure-TypeScript module with three concerns — (1) typed key construction (`K.meta()`), (2) Zod-validated reads with safe fallback (`readWithSchema`), (3) a boot-time migration runner (`runMigrations()`). Install Zod 4 as the validation library. Wire `runMigrations()` into `src/main.tsx` so it executes synchronously BEFORE React mounts, and confirm `halo:v1:meta` is created on first boot with `{ schemaVersion: 1, seededAt: null, appVersion }`.

Purpose: FND-04 and FND-05 are the *single* expensive-to-retrofit pieces of Phase 1. Once feature code starts being written in Phases 2–5, it will reach for direct `localStorage.getItem` unless the discipline is established AND there is a working helper to reach for instead. This plan ships both. The migration runner is empty (we're at v1, nothing to migrate from), but the *infrastructure* must exist so Phase 2 can register `migrations[1] = migrateV1ToV2` without re-thinking the design.

Output:
- `zod` (current major, `^4.x`) installed
- `src/storage/keys.ts` — namespaced key builder
- `src/storage/codec.ts` — `readWithSchema`/`writeJSON`/`removeKey`
- `src/storage/schemas.ts` — `MetaSchema`, `AnonIdSchema`, type exports
- `src/storage/migrations.ts` — `runMigrations()` with empty migrations registry
- `src/storage/index.ts` — barrel export
- `src/main.tsx` calls `runMigrations()` BEFORE `ReactDOM.createRoot(...).render(<App />)`
- A tiny smoke check (script or test file) confirming a fresh `localStorage` produces a `halo:v1:meta` entry after `runMigrations()`

This plan does NOT add the `StorageProvider` React component — Plan 04 owns that. This plan does NOT install or use `nanoid` for the anonymous visitor ID — Phase 6 owns that. The `AnonIdSchema` is defined here only because Phase 6's `getOrCreateAnonymousVisitorId()` will reuse it; defining it now is cheaper than adding a one-line schema later.
</objective>
