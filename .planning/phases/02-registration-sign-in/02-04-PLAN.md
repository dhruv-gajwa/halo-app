<objective>
Build the sessionStorage wizard-draft module that AUTH-07 mandates: every step submission writes the validated form values into `sessionStorage[K.signupDraft()]`, and refreshing/navigating mid-wizard restores the partial state so the user doesn't lose typed input.

This is the *only* module in the codebase allowed to touch `sessionStorage` for the signup draft. Like the storage codec for localStorage (Plan 01-03), centralizing the access points makes it trivial for downstream plans (02-07..09) to call `writeWizardDraftStep('step2', { jobTitle, role, yearsExperience, location })` without re-implementing the read-merge-write dance every step page.

Key design points (locked):
- Backed by `sessionStorage`, not `localStorage` — UI-SPEC AUTH-07 says "sessionStorage so browser refresh / back / forward mid-flow doesn't lose progress; clears on completion or sign-out." sessionStorage is tab-scoped — it survives refresh + back/forward, doesn't leak across tabs, and clears on tab close (an extra privacy win for partial PII).
- Validated against `SignupDraftSchema` (created in Plan 02-02) on every read; corrupt or malformed values fall through to `{}`.
- `writeWizardDraftStep` merges into the existing draft so writing step 3 doesn't wipe steps 1+2.
- `clearWizardDraft()` is the one-call cleanup invoked on (a) successful wizard completion (Plan 02-09's submit handler) and (b) sign-out (Plan 02-05's `signOut`).
- A `hasStep(draft, 'step1')` helper used by Wave 3 step pages to redirect to `/signup` when the user deep-links to a later step without completing step 1.

Output: 1 new auth-domain module, 1 Node smoke test. Pure data layer; no React, no provider, no router wiring.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User input (in-flight signup) → `sessionStorage` | Partial PII (email, name, company) crosses into browser storage before any registration completes. |
| `sessionStorage` → app | sessionStorage values may be inspected/modified via DevTools or a sibling extension; reads must be schema-validated. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-15 | Information disclosure | Wizard PII retained across users on a shared machine | mitigate | sessionStorage is tab-scoped — it clears on tab close. UI-SPEC AUTH-07 + Plan 02-05 + Plan 02-09 explicitly call `clearWizardDraft()` on sign-out AND on successful wizard completion. The combination means a wizard-abandoned tab is the only retention window. Smoke test asserts `clearWizardDraft` empties storage. |
| T-02-16 | Tampering | sessionStorage value | mitigate | Every read goes through `SignupDraftSchema.safeParse`; corrupt or attacker-edited values fall through to `{}` (smoke test verifies both `not-json` and schema-invalid cases). |
| T-02-17 | Information disclosure | Password held in wizard draft | mitigate | Step 1's `password` field DOES land in sessionStorage briefly between step submissions (the draft is per-step + plaintext). This is an accepted trade-off: refreshing during step 1 means the user re-types the password; refreshing during step 2+ retains it so the wizard can complete on resume. UI-SPEC AUTH-07 + AUTH-08 lock this behavior. Mitigations: (a) sessionStorage is tab-scoped, (b) `clearWizardDraft` runs on completion (the moment the password is hashed and the plaintext is no longer needed), (c) sessionStorage clears on tab close. To strengthen further, Plan 02-09's submit handler could overwrite `draft.step1.password` with empty string before the final `clearWizardDraft` — note that as a future enhancement (not in this plan's scope, since this plan owns only the storage module). |
| T-02-18 | Denial of service | sessionStorage quota | accept | A SignupDraft is a few hundred bytes; sessionStorage quota is 5MB per origin. Cannot exhaust. |
</threat_model>
