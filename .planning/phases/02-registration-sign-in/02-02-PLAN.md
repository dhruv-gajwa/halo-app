<objective>
Define the type-system and validation contract that every subsequent Phase 2 plan builds against. This plan creates:

1. **Zod step schemas** (`step1Schema`..`step4Schema`) â€” one per wizard step, fields matching `02-UI-SPEC.md`'s "Field labels" table and error messages matching the "Inline validation errors" table verbatim. Used by RHF resolvers in Wave 3 page tasks (02-07..09).
2. **Zod sign-in schema** (`signinSchema`) â€” email + password with UI-SPEC errors. Used by 02-10.
3. **Zod persistence schemas** (`VisitorSchema`, `WorkspaceSchema`, `SessionSchema`, `SignupDraftSchema`) plus array schemas for the localStorage value shape. Validated on every `readWithSchema` read per CLAUDE.md ("untrusted disk data").
4. **TypeScript types** derived from Zod schemas â€” `Visitor`, `Workspace`, `Session`, `SignupDraft`, plus a `User` alias of `Visitor` so `AuthProvider`'s existing `User` import continues to type-check.
5. **Storage key builders** â€” `K.visitors()`, `K.workspaces()`, `K.session()`, `K.signupDraft()` â€” added to the existing `K` object in `src/storage/keys.ts` so domain-aware code (Plans 02-03, 02-04, 02-05) doesn't hand-type key strings.
6. **Barrel re-exports** so consumers import from `./storage` or `./auth` without reaching into module internals.

Purpose: schemas + types are pure data â€” no React, no DOM, no side effects â€” and they unblock Wave 1's other plans (02-03 password hash + repo, 02-04 wizard sessionStorage) and all of Wave 2/3. Splitting them into separate plans would create artificial Wave 1 dependencies.

Output: 2 new auth-domain files (`types.ts`, `schemas.ts`), 2 modified storage files (`keys.ts`, `schemas.ts`), 1 modified storage barrel. Zero runtime behavior â€” types and schemas only.

Note on AUTH-06 coverage: AUTH-06 says "Each step is validated with React Hook Form + Zod; invalid input blocks advance and shows inline errors." The Zod schemas (this plan) are the Z half of "RHF + Zod" â€” the resolver wiring + UI surfacing lives in the page plans (02-07..10). This plan owns the schema definitions; Wave 3 owns the form integration. The end-to-end success criterion is verified in the integration plan (02-10).
</objective>
