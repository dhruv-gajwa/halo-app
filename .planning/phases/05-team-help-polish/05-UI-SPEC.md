# Phase 5 — UI Design Contract

> Visual and interaction contract for the Team page (`/app/team`), Help list (`/app/help`), Help detail (`/app/help/:slug`), and the cross-page polish + Pendo-readiness audit. Generated by gsd-ui-researcher.

This contract inherits 100% of its design tokens, primitives, and markup conventions from Phase 4 (which inherited from Phase 3 / Phase 2 / Phase 1). Phase 5 extends with two new PENDO_IDS namespaces (`team`, `help`), adds one new nested router child (`/app/help/:slug`), adds one new storage key (`K.teammates(workspaceId)`), and replaces two ComingSoonCard placeholder pages with real implementations. It MUST NOT redefine theme tokens, alter existing PENDO_IDS leaves, change any `src/ui/primitives/*` wrapper contract, bump `SCHEMA_VERSION`, or introduce any Pendo runtime call.

**Delta from Phase 4 — what Phase 5 adds:**
- Two new pages replace placeholders: Team at `/app/team` and Help (list + detail) at `/app/help` + `/app/help/:slug`.
- One new nested route under `/app/help` (D-07) — first `router.tsx` edit since Phase 3 D-01 lock; surgical and additive.
- New data layer for Team: `TeammateSchema` + `teamsRepo` + `K.teammates(workspaceId)` (D-01) + `WorkspaceRoleEnum` (D-02).
- New static module for Help: `src/help/helpArticles.ts` with `faker.seed(N)` pinned generation (D-09); no storage key.
- New seed coordinator: `src/seed/seedAll.ts` exporting `seedDemoData(workspaceId)`; AppLayout swaps `seedIfNeeded` for `seedDemoData` (D-11). Single `meta.seededAt` gates both seeders (D-12); the stamp moves out of `tasksSeed.ts` into `seedAll.ts`.
- `tasksSeed.ts` surgical amendment — picks assignees from just-seeded teammate list (D-04); removes its own tail `meta.seededAt` stamp.
- `src/tasks/assigneeOptions.ts` source swap from `listTasks` to `listTeammates` (D-04). Consumers (Lists filters, TaskFormModal Assignee, Reports Assignee filter) need no edits.
- PENDO_IDS gains `team` and `help` namespaces (D-14, D-15); `PendoId` derives automatically.
- Cross-page polish (UI-01..04): new Team empty state, new Help no-results state, toast/confirm completeness audit, typography/spacing/color consistency pass across all `/app/*` pages.
- Pendo-readiness verification — manual walk-through via `/gsd-verify-phase` (no `DEMO-READY-AUDIT.md` artifact per D-17). User-eye items persist as UAT entries per Phase 4 commit `5195b3b` precedent.

**Delta from Phase 4 — what Phase 5 does NOT add:**
- No new runtime deps. Mantine 9.2.0, `@mantine/hooks` 9.2.0 (`useDebouncedValue` available), `@mantine/dates` 9.2.1, `@mantine/notifications` 9.2.1, `@tanstack/react-table` 8.21.3, `@tabler/icons-react` 3.44.0, `recharts` 3.8.1, `dayjs` 1.11.20, `nanoid` 5.1.11, `@faker-js/faker` 10.4.0 — all already installed. - No SCHEMA_VERSION bump. `K.teammates(workspaceId)` is a pure additive key (D-01).
- No new destructive flows. "Remove member" is deferred (`<deferred>` in 05-CONTEXT.md). UI-03 is satisfied by Phase 4's existing Delete-task and Reset-demo-data confirmations.

## Spacing Scale

Phase 5 inherits the Phase 3 / Phase 4 spacing contract verbatim. The `xs` token (10px) remains excluded from Phase 5 markup. Phase 5 introduces no new spacing exceptions.

| Mantine token | Pixel value | Usage in Phase 5 |
|---------------|-------------|------------------|
| `xs` | **10px** | **NOT USED in Phase 5.** Excluded — 10 is not a multiple of 4. |
| `sm` | **12px** | Team table cell vertical padding (matches Phase 4 Lists/Reports table padding rule); compact label-to-control gap inside the help no-results state; help article row internal label-to-summary gap |
| `md` | **16px** | Default `<Paper p="md">` inner padding for all Phase 5 surface cards; `<Stack gap="md">` in modal form bodies; `<Group gap="md">` in page headers; table-cell horizontal padding; gap between help search input and the article list; gap between detail title/topic/body paragraphs |
| `lg` | **20px** | Vertical gap between Team page header row and the table (or empty state); vertical gap between Help search input and the topic-grouped list; modal footer `mt="lg"` |
| `xl` | **32px** | Vertical gap between Team table and any future section (currently none); gap between Help search bar / topic-list section and the page bottom; section-to-section spacing on Help detail page (title → body → back link) |

**Hard rules (inherited from Phase 3 / Phase 4):**
- Named Mantine tokens (`sm`, `md`, `lg`, `xl`) are the only string spacing values. `gap="xs"` is FORBIDDEN — use `gap={8}` (numeric) for icon-to-label gaps.
- Raw pixel values are FORBIDDEN as CSS props. Every spacing prop references a Mantine token or the approved `gap={8}` numeric exception.
- Exception: Tabler icon `size={18}`, `size={16}`, `size={64}` props — numeric pixels, not tokens.
- Empty-state icon `size={64}` and `stroke={1.2}` are structural exceptions (inherited from Phase 3 / Phase 4 empty-state pattern).
- Team table cell padding: `padding: var(--mantine-spacing-sm) var(--mantine-spacing-md)` — vertical `sm` (12px), horizontal `md` (16px). Implemented via CSS module if Mantine `<Table>` is used, or via TanStack's column-level styling; planner picks. No raw px values.
- Article body paragraph spacing: rendered via `<Stack gap="md">` over an array of `body.split('\n\n')` strings, each as a `<Text size="md">`. No `<br>` or raw CSS line-spacing.

## Typography

Phase 5 inherits the Phase 3 / Phase 4 four-size, two-weight typography contract exactly. No fifth size. No third weight.

| Role | Mantine API | Size | Weight | Line Height | Phase 5 usage |
|------|-------------|------|--------|-------------|---------------|
| Heading | `<Title order={2}>` | **26px** | **600** | 1.35 | Not used in Phase 5 — pages use Sub-heading for titles. |
| Sub-heading | `<Title order={3}>` | **~22px** | **600** | 1.35 | Page titles: `Team`, `Help`. Help article detail title (`{article.title}`). Empty-state heading: `No teammates yet`. Help article-not-found heading: `Article not found`. Modal title: `Invite teammate` |
| Body | `<Text size="md">` (default) | **16px** | **400** | 1.55 | Help article body paragraphs (each paragraph as one `<Text size="md">`); modal description copy (if any); empty-state body copy; help no-results body copy |
| Label | `<Text size="sm">` | **14px** | **400** | 1.45 | Team table column headers (`<Text size="sm" fw={500}>`); teammate name cell; email cell; lastActive relative time cell; help article row title (within the row card); help article summary (`c="dimmed"`); help topic-group section labels (`tt="uppercase" c="dimmed"`); search input field; modal field labels (Mantine auto-applies via `label=` prop); Badge text |

**Phase 5 typography notes:**

1. Team table column headers use `<Text size="sm" fw={500}>` — `fw={500}` semi-emphasis permitted per Phase 3 "sole exception" rule (same treatment as user name in top-bar trigger and Phase 4 Lists/Reports table headers).
2. Help topic-group section labels use `<Text size="sm" tt="uppercase" c="dimmed">{topic}</Text>` — mirrors the Phase 3 KPI label idiom (`size="sm" c="dimmed" tt="uppercase"`). Each topic group has one such label above its article rows.
3. Help article row title uses `<Text size="sm" fw={500}>{article.title}</Text>` — Label role with `fw={500}` semi-emphasis. The summary line below uses `<Text size="sm" c="dimmed">{article.summary}</Text>` — same size, dimmed color, regular weight.
4. Help article detail title uses `<Title order={3}>{article.title}</Title>` — Sub-heading role (~22px, weight 600). The article body paragraphs use `<Text size="md">` — Body role.
5. Help article detail topic uses `<Text size="sm" c="dimmed">{article.topic}</Text>` — Label role, dimmed.
6. Modal title `<Title order={3}>Invite teammate</Title>` — Sub-heading role. Consistent with Phase 4 modal-title rule.
7. Modal email + role labels use Mantine's built-in `label="Email"` / `label="Role"` props — Mantine renders these at Label role (`size="sm" fw={500}` internally). No override needed.
8. The `Invited` badge text uses Mantine `<Badge size="sm">` — auto-resolves to Label role internally.
9. The `Owner` static badge (if chosen over disabled-option mechanism per D-02) uses Mantine `<Badge size="sm" color="indigo" variant="light">Owner</Badge>` — same Label-role internal sizing.
10. No `fw={700}` anywhere in Phase 5 (inherited prohibition).
11. No `order={4}` `<Title>` anywhere in Phase 5 (inherited).

Mechanism B (static Badge for owner row, Select for everyone else):

{teammate.workspaceRole === 'Owner'

      value={teammate.workspaceRole}

### Help page (`/app/help`)

**Page structure:**

```
<Stack gap="lg">
  <Title order={3}>Help</Title>

  <TextInput
    placeholder="Search articles"
    leftSection={<IconSearch size={16} />}
    value={query}
    onChange={(e) => setQuery(e.currentTarget.value)}
    pendoId={PENDO_IDS.help.search}
  />

  {filteredArticles.length === 0
    ? <HelpNoResultsState query={debouncedQuery} onClear={() => setQuery('')} />
    : <TopicGroupedArticleList groups={topicGroups} />
  }
</Stack>
```

**Search behavior (D-08):**
- `useDebouncedValue(query, 150)` from `@mantine/hooks` → `debouncedQuery`.
- Match logic: substring (case-insensitive, whitespace-tolerant) over `title + keywords[].join(' ') + summary`. Body is NOT searched.
- Empty query: render all articles, grouped by topic (no filter active).
- Non-empty query with matches: render matching articles, grouped by topic. Topic groups with zero matching articles after filtering are hidden.
- Non-empty query with zero matches: render `<HelpNoResultsState>`.

**Topic-grouped list layout:**

```
<Stack gap="xl">
  {topicGroups.map((group) => (
    <Stack key={group.topic} gap="md">
      <Text size="sm" tt="uppercase" c="dimmed" fw={500}>{group.topic}</Text>
      <Stack gap="sm">
        {group.articles.map((article) => (
          <HelpArticleRow key={article.id} article={article} />
        ))}
      </Stack>
    </Stack>
  ))}
</Stack>
```

Topic order: render topics in the order they first appear in `HELP_ARTICLES` (which is faker-seeded with `faker.seed(N)`, so order is stable across reloads). Topic-group spacing: `gap="xl"` between groups; `gap="md"` between the topic label and the article row stack; `gap="sm"` between rows within a group.

**Help article row (`<HelpArticleRow>`):**

Each article rendered as a single `<Anchor component={Link} to={`/app/help/${article.slug}`}>` row. Two acceptable visual treatments — planner picks at compose time:

Treatment A — flat anchor list (lighter, default Mantine anchor):
```tsx
<Anchor
  component={Link}
  to={`/app/help/${article.slug}`}
  pendoId={PENDO_IDS.help.article.row}
  data-pendo-article-slug={article.slug}
  underline="hover"
  c="inherit"
>
  <Stack gap={4}>
    <Text size="sm" fw={500}>{article.title}</Text>
    <Text size="sm" c="dimmed">{article.summary}</Text>
  </Stack>
</Anchor>
```

Treatment B — card-style row (`<Paper withBorder p="md" radius="md">` wrapping the anchor content):
```tsx
<Paper component={Link} to={...} withBorder p="md" radius="md" data-pendo-id={PENDO_IDS.help.article.row} data-pendo-article-slug={article.slug}>
  <Stack gap={4}>
    <Text size="sm" fw={500}>{article.title}</Text>
    <Text size="sm" c="dimmed">{article.summary}</Text>
  </Stack>
</Paper>
```

Treatment A is preferred for the polished-list aesthetic; Treatment B is acceptable if planner wants stronger row separation. Both satisfy the design contract. The `data-pendo-article-slug={article.slug}` dynamic attribute is mandatory in both treatments (CLAUDE.md dynamic-list parameterization rule).

`gap={4}` (numeric, 4px) on the inner `<Stack>` is the only Phase 5 spacing exception below `sm` — `gap={4}` is a multiple of 4, satisfies the multiple-of-4 rule, and is the standard tight-stack idiom from the Phase 3 user-menu trigger pattern (`gap={8}` for icon-to-text was the precedent; `gap={4}` for title-to-summary is a tighter variant within the same "numeric multiples of 4" exception class).
