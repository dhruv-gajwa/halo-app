# Phase 4 — UI Design Contract

> Visual and interaction contract for Lists (`/app/lists`), Settings (`/app/settings`), and Reports (`/app/reports`) inside the Phase 3 AppShell. Generated by gsd-ui-researcher.

This contract inherits 100% of its design tokens, primitives, and markup conventions from Phase 3 (which inherited from Phase 2 and Phase 1). Phase 4 extends with three new PENDO_IDS namespaces (`lists`, `settings`, `reports`), adds dark-mode wiring to the existing AppShell, and introduces new UI surfaces (TanStack Table, task modals, settings tabs, Recharts stacked bar). It MUST NOT redefine Mantine theme tokens, alter existing PENDO_IDS leaves, or change any `src/ui/primitives/*` wrapper contract beyond adding new wrappers.

**Delta from Phase 3 — what Phase 4 adds:**
- Dark mode: `MantineProvider defaultColorScheme="light"` changes to `defaultColorScheme="auto"` in App.tsx; `index.html` ColorSchemeScript inline script gains auto-detection logic. The AppShell and all Phase 3 surfaces recolor automatically via Mantine CSS-var tokens — Phase 4 verifies no regressions.
- Three new pages replace ComingSoonCard placeholders: Lists, Settings, Reports.
- New runtime deps to install: `@tanstack/react-table` (^8.x), `@mantine/dates` (^9.x), `dayjs` (^1.x), `@mantine/notifications` (^9.x).
- `@mantine/notifications` must be installed **and** `<Notifications />` mounted inside `MantineProvider` in `App.tsx` before any page fires a toast. If absent, Phase 4's first plan adds the install + mount.
- New primitive wrappers to add in `src/ui/primitives/`: `Checkbox.tsx`, `Textarea.tsx`, `DatePickerInput.tsx`. Whether `Modal`, `MenuTrigger`, or `Tabs` need wrappers is a planner decision — see §New Primitives below.

## Spacing Scale

Phase 4 inherits the Phase 3 spacing contract verbatim. The `xs` token (10px) remains excluded from Phase 4 markup. Phase 4 introduces no new spacing exceptions beyond the Phase 3 approved list.

| Mantine token | Pixel value | Usage in Phase 4 |
|---------------|-------------|------------------|
| `xs` | **10px** | **NOT USED in Phase 4.** Excluded — 10px is not a multiple of 4. |
| `sm` | **12px** | Table cell vertical padding; filter bar label-to-control gap (where needed) |
| `md` | **16px** | Default `<Paper p="md">` inner padding for all Phase 4 surface cards; `<Stack gap="md">` in forms; `<Group gap="md">` in filter bars |
| `lg` | **20px** | Between filter bar and table/chart; Settings tab panel inner top spacing; Danger zone `mt="xl"` exception (see below) |
| `xl` | **32px** | Section-to-section vertical gap within pages; `mt="xl"` separating Danger zone card from theme toggle |

**Danger zone spacing exception (D-17):** The `<Paper>` Danger zone card uses `p="lg"` (not `p="md"`) matching D-17's spec (`p="lg"`). All other `<Paper>` surfaces in Phase 4 use `p="md"`.

**Hard rules (inherited from Phase 3):**
- Named Mantine tokens (`sm`, `md`, `lg`, `xl`) are the only string spacing values. `gap="xs"` is FORBIDDEN — use `gap={8}` (numeric) for icon-to-label gaps.
- Raw pixel values are FORBIDDEN as CSS props. Every spacing prop references a Mantine token or the approved `gap={8}` numeric exception.
- Exception: Tabler icon `size={18}`, `size={16}`, `size={64}` props — numeric pixels, not tokens.
- The empty-state icon `size={64}` is also a structural exception (inherited from Phase 3 empty-state pattern).
- TanStack Table cell padding: implemented via CSS module or Mantine `Table` prop; planner picks approach. If CSS module, use only `padding: var(--mantine-spacing-sm) var(--mantine-spacing-md)` — no raw px values.

## Typography

Phase 4 inherits the Phase 3 four-size, two-weight typography contract exactly. No fifth size. No third weight. Phase 4 uses exactly four font sizes inherited from Phase 3 — no fifth size added.

| Role | Mantine API | Size | Weight | Line Height | Phase 4 usage |
|------|-------------|------|--------|-------------|---------------|
| Heading | `<Title order={2}>` | **26px** | **600** | 1.35 | Page section headings if needed (not expected — pages use sub-heading or label for titles) |
| Sub-heading | `<Title order={3}>` | **~22px** | **600** | 1.35 | Empty-state heading ("No tasks yet", "No tasks match these filters"); Settings tab section sub-labels if needed; Danger zone card heading (`c="red.7"` distinguishes it visually from other Sub-heading instances) |
| Body | `<Text size="md">` (default) | **16px** | **400** | 1.55 | Modal body copy; Danger zone description; Settings intro copy; empty-state body copy |
| Label | `<Text size="sm">` | **14px** | **400** | 1.45 | Table column headers; filter labels; form field labels (Mantine auto-applies via `label=` prop); badge text; tab labels; chart axis ticks (`fontSize: 14` on `<XAxis tick>` and `<YAxis tick>`) |

**Phase 4 typography notes:**

1. Table column headers: use `<Text size="sm" fw={500}>` — the `fw={500}` semi-emphasis is permitted per the "sole exception" rule from Phase 3 (note 5). This is the same weight treatment as the user name in the top-bar trigger.
2. The Danger zone card heading uses `<Title order={3} c="red.7">Danger zone</Title>` — Sub-heading role (~22px, weight 600). The `c="red.7"` color token visually distinguishes this heading from other Sub-heading instances without adding a fifth font size. No `order={4}` appears anywhere in Phase 4.
3. Chart `<XAxis tick={{ fontSize: 14 }} />` and `<YAxis tick={{ fontSize: 14 }} />` — SVG attribute, same as Phase 3. Not a new font size.
4. No `fw={700}` anywhere in Phase 4 (inherited prohibition).
5. Recharts `<Legend>` text: `style={{ fontSize: 14 }}` — align with Label role.
6. Modal title: `<Title order={3}>` (Sub-heading role, ~22px, weight 600). Consistent with Phase 3 empty-state heading level.

## Color

Phase 4 inherits the Phase 3 indigo-on-neutral palette. The only color additions in Phase 4 are:
1. Dark mode token resolution (CSS vars replace hardcoded hex in Recharts props).
2. Badge color mapping (status + priority) — lives in `src/tasks/labels.ts` as `TASK_STATUS_BADGE_COLOR` and `TASK_PRIORITY_BADGE_COLOR`; not new colors, just named uses of existing Mantine palette tokens.
3. Red treatment for destructive actions (Danger zone card, Delete buttons, destructive-confirm modal button) — uses the Phase 3 `red.6` / `color="red"` token already declared.

| Role | Mantine token | Approximate hex (light) | Usage in Phase 4 |
|------|---------------|-------------------------|------------------|
| **Dominant (60%)** | `--mantine-color-body` | **`#ffffff`** (light) / `#1a1b1e` (dark) | Page background; all `<Paper>` surfaces auto-resolve via CSS var in dark mode |
| **Secondary (30%)** | `<Paper withBorder>` surface | Surface **`#f8f9fa`** (light), border **`#dee2e6`** (light) | All new Phase 4 `<Paper withBorder p="md" radius="md">` surfaces: filter bar wrapper (if wrapped), table wrapper, chart wrapper, Settings Danger zone card, task modal body |
| **Accent (10%)** | `indigo.6` / `indigo` | **`#4263eb`** (light) | See "Accent reserved for" exhaustive list below |
| **Destructive** | `red.6` / `color="red"` | **`#fa5252`** | Danger zone card title (`c="red.7"`); Reset button (`color="red" variant="outline"`); Delete confirm button (`color="red"` filled); Delete kebab item label (`color="red"` on `<Menu.Item>`) |
| **Muted text** | `c="dimmed"` | **`#868e96`** (light) | Empty-state body copy; filter bar descriptions; table null values (`—`); Settings help text |
| **Badge: To do** | `indigo.3` | `#a5b4fc` (light) | Status badge for `todo` tasks — consistent with Phase 3 donut segment |
| **Badge: In progress** | `indigo.6` | `#4263eb` (light) | Status badge for `in_progress` tasks |
| **Badge: Done** | `gray.5` | `#adb5bd` (light) | Status badge for `done` tasks |
| **Badge: Low** | `gray.5` | `#adb5bd` (light) | Priority badge for `low` |
| **Badge: Medium** | `yellow.5` | `#fcc419` (light) | Priority badge for `medium` |
| **Badge: High** | `orange.5` | `#ff922b` (light) | Priority badge for `high` |
| **Badge: Urgent** | `red.6` | `#fa5252` (light) | Priority badge for `urgent` |

**Accent (indigo) reserved for — Phase 4 exhaustive additions. Phase 3 list is inherited and not repeated here:**

Phase 4 adds NO new indigo surfaces beyond those declared in Phase 3. Specifically:
- The Stacked bar chart "To do" bars use `indigo.3`; "In progress" bars use `indigo.6` — these are chart-SVG only (same rule as Phase 3 donut segments).
- The "New task" primary button: `variant="filled"` (Mantine default for `primaryColor="indigo"`) — exactly one primary CTA per page section, consistent with Phase 2/3 rule.
- Settings Save buttons: `variant="filled"` (indigo) — exactly one per tab form.
- No new indigo backgrounds on any container, card, or modal surface.

**Forbidden uses of indigo in Phase 4 (inherited + new):**
- Table row hover backgrounds — use Mantine's default table row hover (which is `gray.0` in light, auto-resolves in dark).
- Filter bar container backgrounds.
- Modal overlay background (Mantine default — black at 55% opacity; do not tint with indigo).
- Danger zone card border — uses Mantine `withBorder` (`gray.3` border), not indigo.

**Dark mode color contract:**

Phase 4 is the first phase to verify and ship dark mode. Rules:

1. Every new Phase 4 surface MUST resolve via `var(--mantine-color-*)` tokens — no hardcoded hex in component bodies.
2. Recharts `fill` and `stroke` props that used named token strings in Phase 3 (`"indigo.6"`, `"gray.2"`) MUST be converted to CSS variable resolution. The Mantine `parseThemeColor` utility or `useMantineTheme()` hook can resolve token strings to hex at render time. Recommended pattern for the stacked bar chart:

```tsx
// Resolution approach — use Mantine theme.colors lookup
import { useMantineTheme } from '@mantine/core'
const theme = useMantineTheme()
const colorTodo      = theme.colors.indigo[3]   // resolves for current scheme
const colorInProgress = theme.colors.indigo[6]
const colorDone      = theme.colors.gray[5]
const colorGrid      = theme.colors.gray[2]
```

3. `indigo.6` in both light and dark schemes resolves to the same RGB — dark contrast check: in dark mode `indigo.6` (#4263eb) on a dark surface (#1a1b1e or similar) has sufficient contrast for bars in a chart but should be verified manually. If contrast is poor, use `indigo.4` or `indigo.5` for dark scheme. Use the `useComputedColorScheme` hook to conditionally swap:

```tsx
const computed = useComputedColorScheme('light')
const colorInProgress = computed === 'dark' ? theme.colors.indigo[4] : theme.colors.indigo[6]
```

4. `CartesianGrid` stroke: use `var(--mantine-color-gray-2)` (same as Phase 3 Area chart) — auto-resolves in dark.
5. Verification step for every Phase 4 page: manually toggle theme via the Preferences tab SegmentedControl and confirm no hardcoded colors leak through.

### Lists page (`/app/lists`)

**Page structure:**

```
<Stack gap="lg">
  {/* Page header row */}
  <Group justify="space-between" align="center">
    <Title order={3}>Tasks</Title>
    <Button
      variant="filled"
      leftSection={<IconPlus size={16} />}
      pendoId={PENDO_IDS.lists.newTaskButton}
      onClick={openCreateModal}
    >
      New task
    </Button>
  </Group>

  {/* Filter bar */}
  <Group gap="md" data-pendo-id={PENDO_IDS.lists.filter.bar}>
    <Select label="Status"   pendoId={PENDO_IDS.lists.filter.status}   ... />
    <Select label="Priority" pendoId={PENDO_IDS.lists.filter.priority} ... />
    <Select label="Assignee" pendoId={PENDO_IDS.lists.filter.assignee} ... />
  </Group>

  {/* Table — or empty state */}
  {emptyForever
    ? <ListsEmptyState onCreateClick={openCreateModal} />
    : filteredEmpty
      ? <FilteredEmptyState onClearFilters={resetFilters} />
      : <TaskTable ... />
  }
</Stack>
```

**"New task" button placement:** Top-right of the page header row (`justify="space-between"` Group), NOT inside a table card header. Rationale: guide anchoring symmetry — the button is always visible as a stable page-level element regardless of table state. `data-pendo-id` from `PENDO_IDS.lists.newTaskButton`.

**Filter bar:** `<Group gap="md">` containing three Mantine `<Select>` controls. No `<Paper withBorder>` wrapper on the filter bar itself — it sits directly in the page Stack, not inside a card. The three Selects are narrow (`w={160}`) to fit side by side without wrapping at standard desktop widths. Options:

| Filter | Options | Default | clearable |
|--------|---------|---------|-----------|
| Status | All / To do / In progress / Done | All | `false` — "All" is the cleared state |
| Priority | All / Low / Medium / High / Urgent | All | `false` |
| Assignee | All / \{distinct assignees from tasks} | All | `false` |

**Modal footer button order:**

```
<Group justify="space-between" mt="lg">
  {/* Left — destructive, edit mode only */}
  {mode === 'edit' && (
    <Button color="red" variant="subtle" pendoId={PENDO_IDS.lists.modal.delete} onClick={openDeleteConfirm}>
      Delete task
    </Button>
  )}

  {/* Right — always present */}
  <Group gap="md" ml="auto">
    <Button variant="default" pendoId={PENDO_IDS.lists.modal.cancel} onClick={close}>
      Discard
    </Button>
    <Button
      variant="filled"
      type="submit"
      disabled={!form.formState.isDirty && mode === 'edit'}
      pendoId={PENDO_IDS.lists.modal.save}
    >
      {mode === 'create' ? 'Create task' : 'Save changes'}
    </Button>
  </Group>
</Group>
```

Rules:
- Discard is always left of Save (left → right: Discard / Save).
- Delete is left-aligned (destructive actions left, primary right — visual quarantine).
- In create mode, Delete button is absent entirely.
- Closing the edit modal to open the delete confirm: close the edit modal first (`close()`), then open the delete confirm. Avoid modal stacking.
- Save button disabled only in edit mode when form is not dirty. In create mode, Save is always enabled (Zod validates on submit).

**RHF configuration:** `mode: 'onSubmit'` (Phase 2 lock). Errors appear after first submit attempt, then update on subsequent changes. Error idiom: Mantine `error=` prop on each input, populated by `form.formState.errors.<field>.message`.

## CSV Export Contract

**Filename:** `halo-tasks-YYYY-MM-DD.csv` where `YYYY-MM-DD` = `dayjs().format('YYYY-MM-DD')` (browser local time).

**Column header row (verbatim — matches visible table column labels):**

```
Title,Status,Priority,Assignee,Due date,Completed at
```

**Field formatting for CSV:**
- Status: use `TASK_STATUS_LABELS[task.status]` (e.g., `"To do"`, `"In progress"`, `"Done"`).
- Priority: use `TASK_PRIORITY_LABELS[task.priority]` (e.g., `"Low"`, `"Medium"`).
- Assignee: `task.assignee?.name ?? ''` (empty string, not `—`, in CSV — em-dash is a display-only convention).
- Due date: `task.dueDate ? dayjs(task.dueDate).format('YYYY-MM-DD') : ''` — ISO date for CSV (machine-readable), not `'MMM D, YYYY'`.
- Completed at: `task.completedAt ? dayjs(task.completedAt).format('YYYY-MM-DD') : ''`.

**RFC 4180 quoting:** Double-quote any field that contains a comma, newline, or double-quote. Escape internal double-quotes by doubling (`"` → `""`). Title field is most likely to need quoting.

**Implementation pattern (no CSV library):**
```ts
function toCsv(rows: Task[]): string {
  const headers = ['Title', 'Status', 'Priority', 'Assignee', 'Due date', 'Completed at']
  const lines = rows.map((t) => [
    quote(t.title),
    quote(TASK_STATUS_LABELS[t.status]),
    quote(TASK_PRIORITY_LABELS[t.priority]),
    quote(t.assignee?.name ?? ''),
    t.dueDate ? dayjs(t.dueDate).format('YYYY-MM-DD') : '',
    t.completedAt ? dayjs(t.completedAt).format('YYYY-MM-DD') : '',
  ].join(','))
  return [headers.join(','), ...lines].join('\r\n')
}
```

**Download sequence:**
1. `const blob = new Blob([csv], { type: 'text/csv;charset=utf-8' })`
2. `const url = URL.createObjectURL(blob)`
3. Create ephemeral `<a href={url} download={filename}>`, click it, then `URL.revokeObjectURL(url)`.
