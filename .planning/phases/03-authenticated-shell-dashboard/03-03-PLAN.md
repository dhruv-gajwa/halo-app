<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| faker output → localStorage write | faker generates synthetic data; trustable but must conform to TaskSchema or downstream readers via `tasksRepo.listTasks` would discard the entire blob |
| meta.seededAt timing → user mutations | If gate misfires (seededAt is null but tasks already exist), Phase 4 user-created tasks could be overwritten on next mount |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-03-01 | Tampering | seeded data shape drift vs TaskSchema | mitigate | Defensive `TasksArraySchema.safeParse(tasks)` call inside `generateTasks` before write — if a future schema edit makes the seeded shape invalid, this fails loudly at seed time instead of silently producing a corrupt blob that `listTasks` then discards as `[]`. |
| T-03-03-02 | Tampering | seededAt-null + tasks-already-present race | mitigate | GATE 2 (defensive): `existing.length > 0` check before write. If a DevTools edit or external script wrote tasks before seededAt was stamped, the seeder skips re-seeding instead of clobbering. Documented in the file-header docblock. |
| T-03-03-03 | Information Disclosure | faker-generated names / emails in localStorage | accept | No PII; faker output is synthetic. The CLAUDE.md Out-of-Scope explicitly forbids real PII in this app (DemoBanner: "Demo data only — never enter real credentials"). |
| T-03-03-04 | Denial of Service | localStorage QuotaExceededError on 60-task write | accept | `writeJSON` is non-fatal (storage/codec.ts:44-50 — logs warning, never throws). Even with description paragraphs ~2KB each, 60 tasks * 3KB ≈ 180KB — well under the 5MB per-origin localStorage cap. No additional mitigation required. |
</threat_model>
